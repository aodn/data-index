# Size-threshold routing in FileFetcher

The optimal fetch strategy for a NetCDF file depends on its size: for small files it is faster to download the whole file to disk and read it sequentially; for large files, targeted byte-range reads via fsspec block cache (S3XarrayHandle) avoid downloading data that the extractor will never use. The planned `ThresholdFileFetcher` encapsulates both strategies behind the single `FileFetcher` protocol, routing each URI to either a `DiskXarrayHandle` or an `S3XarrayHandle` based on a configurable `size_threshold_bytes`. The `Batch` already carries file sizes, so the routing decision is made before any I/O begins. Alternatives considered: two separate fetcher implementations with an external router (splits the routing decision away from the fetcher, complicating the pipeline wiring); a single fetcher that always downloads (too slow for large files); a single fetcher that always uses byte-range reads (too slow for small files due to per-request overhead).
